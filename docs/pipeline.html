Price List Extraction Pipeline

From a supplier file to canonical, audit-traced rows. This page shows where Opus 4.7, vision, agents, tools, skills, and deterministic code each fit — and where they explicitly do not.

per-document workflow · evaluator–optimizer pattern · code orchestration with LLM leaf decisions

The six things you are looking at

Every box in the flow below is one of these. The colored stripe on the left of each box tells you which.

Opus 4.7
The model. Every LLM call in this pipeline is Opus 4.7 (1M context). Used in three roles: generator (extractor, normalizer), validator (adversarial), and tiebreaker (final automated arbiter). Same weights, different system prompts and tools.
Vision
Page images sent as input. Used for Stage 1 extraction and the validators / tiebreakers that ground claims against the page. Not used downstream: once Stage 1's rows have been pixel-verified, that text is the authority for Stage 2.
Agents
Bounded, role-specific Opus calls with adversarial system prompts. Validators are scoped to ≤100 rows each. Tiebreakers are scoped to one dispute at a time. Each has a separate prompt and tool schema from the generator. They run in parallel and are stateless.
Tools
Tool-use schemas binding strict JSON output. tool_choice is set to a specific tool, so the model must fill the schema. One tool per role: submit_extraction, submit_pixel_verdicts, submit_completeness_verdict, submit_normalization, submit_blind_matches, submit_justification_verdicts, submit_stage1_tiebreaker_decision, submit_stage2_tiebreaker_decision.
Deterministic code
Python orchestration. Format detect, page rendering, chunking, validator dispatch, verdict aggregation, dispute detection, repair distillation, audit-record assembly, commit. All control flow lives here. No LLM decides what step runs next.
Skills
User-installable capability extensions in Claude Code / Agent SDK. Not central to this pipeline. If you deploy on Claude Code or Agent SDK, the orchestrator can be exposed as a slash command (e.g., /process_pricelist). For pure Anthropic API deployment, skills don't apply. See the note at the bottom for the legitimate places they fit.

Pipeline (per document)

Top to bottom. Each box names its category; the badges name the modalities and contracts in play. Loops and conditional escalations are called out at their decision points.

00 · File arrives

PDF or image (xlsx out of scope per current decision). Filename is preserved as a signal.

01 · Format detect & preprocess Code

Detect MIME, render PDF pages to images at vision-quality DPI, hash inputs, register replay pointers.

infile bytes, filename outpage images, content_hash, document_id
02 · Stage 1 — Extract Opus 4.7 + Vision Tool Cached

Generator role. Reads page images, identifies supplier and effective dates, extracts every line item verbatim. No catalog awareness. Layout-describe pass first, then extraction.

inpage images (cached), filename, today toolsubmit_extraction outsupplier, effective_dates, dates_found, layout_description, items[], uncertainties[]
03 · Chunk for validators Code

Slice extracted rows into chunks of ≤100 for Validator A. Slice pages (or page regions) for Validator B. Same item_ids across both so verdicts join cleanly.

04 · Parallel
Validator A — Pixel verification Agent + Vision Tool

Adversarial. For each extracted row, locate it on the page and confirm every field is literally printed there. Catches hallucinations and inferred-not-extracted fields.

inpage images, ≤100 row chunk toolsubmit_pixel_verdicts outper-row verdict + page_quote
Validator B — Completeness audit Agent + Vision Tool

Adversarial. Independently scans the page; reports rows visible but not extracted, rows extracted that are actually section headers, missed sold-out flags, and supplier/date corrections.

inpage (or region) image, raw_row_text list toolsubmit_completeness_verdict outmissed_rows, misclassified_rows, missing_flags, supplier_check, dates_check
05 · Aggregate verdicts & detect disputes Code

Join A and B by item_id. Compute pass/fail per pass criteria. Detect tiebreaker triggers: S1-HALLUCINATION, S1-PHANTOM_MISS, S1-FIELD_DISPUTE.

06 · Conditional — only on disputes
Stage 1 Tiebreaker Agent + Vision Tool

Re-derives from the page from scratch, then picks a verdict. Cap: if >15% of rows trigger tiebreakers, the document escalates to human review without further calls.

inpage images, dispute pattern, all party positions toolsubmit_stage1_tiebreaker_decision outdecision (KEEP / DROP / ADD / FIX / INCONCLUSIVE)
07 · Decide Code
passcontinue to inter-stage gate (08)
repairdistill issues → loop back to 02 with <prior_issues> (max 2 repair iterations)
humanexit to review queue; audit record marked needs_human_review
08 · Inter-stage gate Code

Apply supplier glossary expansions to raw rows. Run heuristic checks (typical-row-count for this supplier, etc.). Surface concerns into pre_stage2_warnings.

Stage 2 boundary — vision is no longer the source of truth
09 · Stage 2 — Normalize against catalog Opus 4.7 Tool Catalog cached

Generator role. Maps each verbatim raw row to a canonical product_id (or null). Catalog (717 rows) lives in the cached system block — written once, hit free thereafter.

incatalog (cached), raw rows from accepted Stage 1 iteration toolsubmit_normalization outper-row mapping, confidence, reasoning, alternatives
10 · Chunk for normalization validators Code

Slice mappings into chunks of ≤100. Same chunks feed C and D so their verdicts can be joined per item_id.

11 · Parallel
Validator C — Blind re-match Agent Tool Catalog cached

Adversarial. Independently picks the best catalog match for each raw row, without seeing Stage 2's answer. Code compares afterward; agreement vs disagreement drives next step.

incatalog (cached), ≤100 raw rows toolsubmit_blind_matches outindependent best_product_id + ranked alternatives
Validator D — Justification audit Agent Tool Catalog cached

Adversarial. Sees Stage 2's pick and argues against it: checks commodity, variety, and pack/size alignment independently. Catches "right answer for wrong reason."

incatalog (cached), ≤100 mappings toolsubmit_justification_verdicts outverdict + dimension_checks + argument_against
12 · Aggregate verdicts & detect disputes Code

Join Stage 2, C, and D by item_id. Detect tiebreaker triggers: S2-MATCH_DISAGREEMENT, S2-DIMENSION_FAILURE, S2-NULL_DISPUTE.

13 · Conditional — only on disputes
Stage 2 Tiebreaker Agent Tool + Vision (escalation)

Re-derives the catalog match from scratch. Optional page-region image attached for ambiguous pack/size cases. Batched up to ~50 disputes per call (catalog stays cached).

incatalog (cached), disputes, optional page region toolsubmit_stage2_tiebreaker_decision outdecision (ADOPT_STAGE2 / ADOPT_C / ADOPT_D / NEW_PICK / NULL / INCONCLUSIVE)
14 · Decide Code
passcontinue to commit (15)
repairdistill issues → loop back to 09 with <prior_issues> (max 2 repair iterations)
humanexit to review queue
15 · Assemble final output & commit Code

Build final.canonical_rows[] with full provenance per row (extracted_in_iter, validator_a verdict, validator_b coverage, normalized_in_iter, validator_c agreement, validator_d verdict, tiebreaker reference). Persist audit record. Set status.

outcommitted audit record + canonical rows ready for downstream consumers

Audit record (written by code throughout)

Every box in the flow above appends to the per-document audit record. The orchestrator owns this record; LLMs never write to it directly. It is append-only, content-addressed, and replayable.

A · Identity & versions
audit_id, document_id, content_hash, schema/pipeline/prompt versions, models per role.
written at: 00–01
B · Preprocessing
Page image hashes, paths, warnings, preprocessing latency.
written at: 01
C · Stage 1 iterations
Per iteration: prompt_hash, full request_blob, raw response, validator A/B runs, aggregated verdict, repair feedback.
written at: 02–07
D · Inter-stage
Glossary applied, heuristic concerns surfaced.
written at: 08
E · Stage 2 iterations
Catalog version, per-iteration prompts/responses, validator C/D runs, aggregated verdict.
written at: 09–14
F · Tiebreakers
Each tiebreaker invocation: trigger, party positions, decision, cost.
written at: 06, 13
G · Final output
canonical_rows with per-row provenance chain back to page region.
written at: 15
H · Metrics
Total cost, tokens (incl. cache read/write), API call count, iteration count, row counts, wall clock.
written at: 15
I · Lifecycle
status, committed_at/by, human_overrides[], replay_pointers.
written at: 15+

Where skills fit (and don't)

Skills are not in the pipeline above. The pipeline is a deterministic Python workflow that calls the Anthropic Messages API. Skills are a Claude Code / Agent SDK concept — user-installable capability extensions exposed as slash commands or model-discoverable abilities.

Three legitimate places where skills could attach if you deploy the orchestrator inside Claude Code or the Agent SDK:

  • Operator-facing entry point — the orchestrator exposed as /process_pricelist <file-or-folder> so a buyer can trigger a run from a Claude Code session.
  • Supplier glossary maintenance — /glossary add, /glossary review for human-curated abbreviation expansions that feed Stage 2.
  • Audit-record exploration — /audit show <document_id> to surface a record for review.

For a pure API deployment (FastAPI worker, Lambda, cron job), skills don't apply at all — the orchestrator is your Python code, not a skill.
